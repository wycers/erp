## Context

当前材料管理页面 (`src/routes/erp/materials/+page.svelte`) 使用简单的文本输入框让用户输入图片 URL。用户需要：
1. 将图片上传到第三方图床
2. 复制生成的 URL
3. 粘贴到表单中

这个流程繁琐且容易出错，特别是对于非技术用户。项目目前没有任何文件上传功能，需要从零实现。

## Goals / Non-Goals

**Goals:**
- 提供直观的图片上传界面，支持点击选择或拖拽上传
- 实现图片预览功能，让用户可以查看已上传的图片
- 支持删除和替换图片
- 保持 API 简单，便于复用到其他需要图片的地方

**Non-Goals:**
- 不实现图片编辑功能（裁剪、缩放等）
- 不实现多图上传（本次只处理单张图片）
- 不实现 CDN 集成或图片优化处理
- 不迁移云存储（先用本地文件系统）

## Decisions

### 1. 存储方案：本地文件系统

**选择**: 将图片存储在 `static/uploads/` 目录

**理由**:
- 简单直接，无需配置外部服务
- 适合当前开发阶段和小规模使用
- 图片可通过 `/uploads/xxx.jpg` 直接访问

**备选方案**:
- 云存储 (S3/R2): 更适合生产环境，但增加复杂度和成本
- 数据库 BLOB: 性能差，不推荐

### 2. 上传 API：独立 API Endpoint

**选择**: 创建 `/api/upload/image` POST endpoint

**理由**:
- 与表单提交分离，支持即时上传和预览
- 返回图片 URL，前端可直接使用
- 便于添加文件验证和错误处理

**备选方案**:
- 表单一起提交: 用户体验差，无法即时预览

### 3. 前端组件：可复用的 ImageUploader 组件

**选择**: 创建 `$lib/components/ui/image-uploader/` 组件

**理由**:
- 遵循项目现有的 UI 组件结构
- 可复用于其他需要图片上传的场景
- 封装上传逻辑，使用简单

### 4. 文件命名：UUID + 原始扩展名

**选择**: 使用 `{uuid}.{ext}` 格式命名文件

**理由**:
- 避免文件名冲突
- 防止路径遍历攻击
- 保留原始文件类型

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 本地存储不适合多实例部署 | 后续可迁移到云存储，API 接口保持不变 |
| 大文件上传影响性能 | 限制文件大小（如 5MB），添加客户端验证 |
| 恶意文件上传 | 严格验证 MIME 类型，只允许图片格式 |
| 磁盘空间耗尽 | 监控存储使用，考虑定期清理未使用图片 |
| 删除图片时文件残留 | 当前暂不处理，后续可添加清理任务 |
